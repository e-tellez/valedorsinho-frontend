This guide covers everything you need to run Valedorsinho locally and contribute new Adyen features.

## Quick Start (Mock Mode — no backend required)

The fastest way to run the full frontend without setting up the FastAPI backend or a real Adyen account.

```bash
# 1. Clone and install
git clone https://github.com/e-tellez/etellez-workspace.git
cd etellez-workspace
npm install

# 2. Create .env.local from the example
cp .env.example .env.local

# 3. Enable mock mode — edit .env.local and set:
#    NEXT_PUBLIC_MOCK_API=true
#    NEXT_PUBLIC_SUPABASE_URL=https://xxxx.supabase.co   ← your Supabase project
#    NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...

# 4. Start the dev server
npm run dev
```

Open [http://localhost:3000/valedorsinho](http://localhost:3000/valedorsinho).

**How mock mode works:**  
When `NEXT_PUBLIC_MOCK_API=true`, a [Mock Service Worker](https://mswjs.io/) intercepts every `/api/*` fetch in the browser and returns realistic stub responses — no FastAPI instance needed. All Adyen flows (checkout, terminal payments, fleet, webhooks) return pre-canned data defined in `src/mocks/handlers.ts`.

> The MSW service worker file lives at `public/mockServiceWorker.js`. It is generated by `npx msw init public/` and committed to the repo — no extra step needed.

## Full Stack Mode (real backend + real Adyen account)

Use this when you want to run actual Adyen API calls.

### 1. Supabase Setup

1. Create a free project at [supabase.com](https://supabase.com)
2. Copy **Project URL** and **anon key** from Project Settings → API
3. Run the following SQL in the Supabase SQL Editor:

```sql
-- User profiles table (stores role)
CREATE TABLE IF NOT EXISTS profiles (
  id     uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  role   text NOT NULL DEFAULT 'im'
);

ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users read own profile" ON profiles FOR SELECT USING (auth.uid() = id);

-- Auto-create profile on signup (assigns 'im' role by default)
CREATE OR REPLACE FUNCTION handle_new_user()
RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  INSERT INTO profiles (id, role) VALUES (NEW.id, 'im');
  RETURN NEW;
END;
$$;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE PROCEDURE handle_new_user();

-- Adyen credentials table
CREATE TABLE IF NOT EXISTS adyen_configs (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  api_key          text NOT NULL DEFAULT '',
  client_key       text NOT NULL DEFAULT '',
  merchant_account text NOT NULL DEFAULT '',
  created_at       timestamptz DEFAULT now(),
  updated_at       timestamptz DEFAULT now(),
  UNIQUE(user_id)
);

ALTER TABLE adyen_configs ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users manage own config" ON adyen_configs FOR ALL
  USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);
```

4. In Authentication → URL Configuration, set **Site URL** to `http://localhost:3000` and add `http://localhost:3000/valedorsinho/auth/callback` to **Redirect URLs**.

### 2. FastAPI Backend

The backend is at a separate private repo. Check with the project owner for access or use mock mode above.

When running locally:

```bash
# In the valedorsinho FastAPI repo:
uvicorn main:app --reload --port 8000
```

Set `VALEDORSINHO_API_URL=http://localhost:8000` in `.env.local`.

### 3. Start the Frontend

```bash
npm run dev
```

## How to Add a New Adyen Feature

Follow these steps to contribute a new Adyen integration (e.g. a new payment method demo, a new terminal flow, a new tool).

### Step 1 — Pick the right epic branch

| Feature type | Epic branch |
|---|---|
| Checkout, payment methods, payload tools | `valedorsinho/epic/digital` |
| Terminal payments, fleet, NFC | `valedorsinho/epic/unified-commerce` |
| Setup, webhooks, management API | `valedorsinho/epic/tools-and-setup` |
| Theme, layout, shared components | `valedorsinho/epic/global-ui` |
| Code cleanup / refactoring | `valedorsinho/epic/code-refactor` |

### Step 2 — Sync and create a feature branch

```bash
git fetch origin
git checkout valedorsinho/epic/<name>
git merge origin/valedorsinho/epic/<name> --no-edit
git checkout -b valedorsinho/feature/<your-feature-name>
```

### Step 3 — Create the route

```
src/app/valedorsinho/<feature-name>/
  page.tsx        # Route entry point
  FeatureDemo.tsx # Client component with the actual UI (if interactive)
```

**Server component (`page.tsx`) — typical pattern:**
```tsx
import type { Metadata } from "next";
import PageHeader from "@/components/adyen/shared/PageHeader";
import FeatureDemo from "./FeatureDemo";

export const metadata: Metadata = { title: "Your Feature – Valedorsinho" };

export default function FeaturePage() {
  return (
    <div className="w-full max-w-[800px]">
      <PageHeader title="Your Feature" subtitle="What it demonstrates." backHref="/valedorsinho" />
      <FeatureDemo />
    </div>
  );
}
```

### Step 4 — Add a DashCard entry to the dashboard

In `src/app/valedorsinho/page.tsx`, add a `<DashCard>` to the appropriate section (Digital, Unified Commerce, or Additional Tools):

```tsx
<DashCard
  href="/valedorsinho/<feature-name>"
  icon={<YourIcon className="w-full h-full" />}
  iconClass="bg-gray-100 dark:bg-[#0a0f1e] text-primary dark:text-primary"
  title="Your Feature"
  description="One-line description."
/>
```

### Step 5 — Add API calls using the shared wrappers

Always use `apiFetch` / `apiGet` / `apiPost` / `apiPut` from `src/lib/adyen/api.ts` — never raw `fetch()`. These wrappers automatically attach the Supabase JWT.

```ts
import { apiGet, apiPost } from "@/lib/adyen/api";

const data = await apiGet<MyType>("/api/your-endpoint", { param: "value" });
const result = await apiPost<MyResult>("/api/your-endpoint", { body: "data" });
```

### Step 6 — Add a mock handler

In `src/mocks/handlers.ts`, add your endpoint to the relevant section so contributors can run without a backend:

```ts
http.get("/api/your-endpoint", async () => {
  await delay(200);
  return HttpResponse.json({ your: "mock data" });
}),
```

### Step 7 — Display API calls with ApiCallPanel

Use `ApiCallPanel` to render a fixed side-panel that lists all API calls made by a page. Never render `ApiCallCard` entries inline — always go through `ApiCallPanel`. The caller specifies which side of the screen to pin it to.

```tsx
import ApiCallPanel from "@/components/adyen/shared/ApiCallPanel";
import { ApiCallEntry } from "@/components/adyen/shared/ApiCallCard";

// In the component:
const [apiCalls, setApiCalls] = useState<ApiCallEntry[]>([]);

// Wrap each apiGet / apiPost with timing and push an entry:
const t0 = Date.now();
const result = await apiPost<MyResult>("/api/your-endpoint", requestBody);
setApiCalls(prev => [...prev, {
  method: "POST",
  endpoint: "/api/your-endpoint",
  direction: "merchant→adyen",
  request: requestBody,
  response: result,
  statusCode: 200,
  latencyMs: Date.now() - t0,
  timestamp: new Date().toISOString(),
}]);

// In JSX — place after the main content div:
<ApiCallPanel side="right" calls={apiCalls} />
// or side="left" — the panel is hidden automatically when calls is empty.
```

> **Never use `PreviewCard` for API call data.** `PreviewCard` is for non-API-call raw JSON display only.

### Step 8 — Update the API contract

If your feature adds or modifies backend endpoints, update `docs/api-contracts/valedorsinho.md` using the `/sync-contract` workflow.

### Step 9 — Commit and open a PR

```bash
git add .
git commit -m "feat(<feature-name>): add <short description>"
git push origin valedorsinho/feature/<your-feature-name>
```

Open a PR into your parent epic branch. See `CONTRIBUTING.md` for PR naming and body template.

## Adyen Golden Rules

> Never use CSS or HTML tricks to control Adyen UI. Always use native SDK properties and callbacks.

| Need | Correct approach |
|---|---|
| Hide pay button on stored cards | `paymentMethodsConfiguration.storedCard.showPayButton: false` |
| Hide security code on stored cards | `paymentMethodsConfiguration.storedCard.hideCVC: true` |
| Rename pay button | Top-level `translations: { "en-US": { payButton: "..." } }` |
| Remove stored method | `onDisableStoredPaymentMethod(id, resolve, reject)` callback |

If no native SDK property exists for a use case → split the UI architecturally. Never DOM/CSS workarounds.
